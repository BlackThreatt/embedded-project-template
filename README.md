# embedded-project-template

A reusable, well-structured template for embedded C projects inspired by @artfulbytes. Clone this repository to instantly scaffold a new embedded project with a working build system, organized source layout, and driver/test separation baked in.

---

## Project Structure

```
embedded-project-template/
├── build/              # Compiled output (object files, binaries) — git-ignored
├── docs/               # Project documentation, datasheets, design notes
├── linker/             # Linker file and SVD if needed
├── external/           # Third-party libraries and dependencies (submodules or vendored)
├── src/
│   ├── app/            # Application-level logic (state machines, tasks, business logic)
│   ├── common/         # Shared utilities, typedefs, macros, and helpers
│   ├── drivers/        # Hardware abstraction / peripheral drivers (GPIO, UART, SPI, etc.)
│   ├── test/           # Unit tests and hardware-in-the-loop test code
│   └── main.c          # Application entry point
├── LICENSE             # Project license
├── Makefile            # Build system
└── README.md           # This file
```

## Getting Started

### Prerequisites

- arm-none-eabi-*, gcc or other compilers can be used after modifying the `Makefile`
- `make`
- `binutils-arm-*` in order to use `size` and `objdump`

### Clone the Template

```bash
git clone https://github.com/BlackThreatt/embedded-project-template.git my-new-project
cd my-new-project
```

If you want to start fresh without the template's git history:

```bash
rm -rf .git
git init
git add .
git commit -m "Initial commit from template"
```

### Build

```bash
make        # Build the project
make bin    # Generate project firmware in binary
make hex    # Generate project firmware in hexadecimal 
make clean  # Remove build artifacts
``` 
The compiled output is placed in the `build/` directory.

## Directory Details

### `src/app/`
High-level application code. This is where your main program logic lives — state machines, task schedulers, application configuration, and feature modules. Keep this layer hardware-agnostic where possible.

### `src/common/`
Shared code used across the project: type definitions (`types.h`), utility macros, ring buffers, error codes, and other reusable building blocks.

### `src/drivers/`
Hardware-specific code. Each peripheral or IC gets its own source/header pair (e.g. `uart.c`, `spi.c`, `gpio.c`). Drivers expose a clean API to the rest of the codebase and isolate hardware dependencies.

### `src/test/`
Test code, unit tests, integration tests, or hardware-in-the-loop stubs. Keep tests independent of `app/` logic where possible to allow off-target testing.

### `external/`
Third-party code such as CMSIS, FreeRTOS, or vendor HAL libraries. Add these as git submodules or vendor them directly. Do not modify external code, patch via wrappers in `src/common/` or `src/drivers/` if needed.

### `docs/`
Project documentation: architecture notes, pin mappings, datasheets, and anything else relevant to understanding or maintaining the project.

### `linker/`
Linker scripts `.ld` for the target MCU that define memory regions and section placement. SVD files `.svd` that describe the device's peripheral register map enabling live inspection during debugging. 

### `build/`
Auto-generated by `make` to contain build artifacts and object files generated during compilation. This directory is excluded from version control via `.gitignore`.

## Makefile

The included `Makefile` provides a ready-to-use build system. Key targets:

| Target        | Description                          |
|---------------|--------------------------------------|
| `make`        | Build the full project                 |
| `make clean`  | Remove all build artifacts             |
| `make flash`  | (TODO) Flash to target MCU             |
| `make test`   | (TODO) Run unit tests                  |

Edit the top of the `Makefile` to set your target MCU, compiler toolchain, and linker script.

---

## Adapting This Template
This template was made to be MCU agnostic, to adapt it following these steps:
1. Update the `Makefile` with your toolchain and target-specific flags.
2. Add your MCU's startup files and linker script (`.ld`) to the project root or a dedicated `ld/` directory.
3. Add external dependencies to `external/` (preferably as git submodules).
4. Start writing application logic in `src/app/` and hardware drivers in `src/drivers/`.
